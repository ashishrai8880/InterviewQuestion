2. Describe project which you have worked on ?

Ans ==> One of the most challenging issues I worked on was in a collaborative layout design application where multiple users
could access the same design file, but only one user was allowed to edit it at a time.

To enforce this, whenever a user opened a file and clicked "Edit", the backend would acquire a lock by storing the file 
information in Redis with a TTL. If another user attempted to edit the same file, the system would prevent it and show that
the file was already locked.

The challenge was that we started receiving occasional reports from users saying that they were unable to edit a file even 
though nobody else was working on it. The issue was very difficult to reproduce because it happened only once or twice a 
week in production.

I began investigating by adding detailed logging on both the backend and frontend. I analyzed WebSocket connection events, 
lock creation requests, and Redis entries. After several days of monitoring, I discovered that the root cause was 
intermittent network disruptions.

What was happening was that a user would successfully acquire a lock, but due to a temporary network interruption, the 
WebSocket connection would disconnect unexpectedly. When the client automatically reconnected, it would send another lock 
request. However, the original lock entry was still present in Redis because its TTL had not expired yet. As a result, the 
server treated the request as if another user had already locked the file and rejected it, even though it was actually the 
same user trying to regain access.

To solve this, I modified the locking logic on the backend. Instead of only checking whether a lock existed, I also 
validated the ownership of the lock. If the same user attempted to reacquire the lock after reconnection, the system would 
allow it and refresh the lock instead of rejecting the request.

After deploying the fix, the issue was completely resolved, and users no longer experienced random lock failures. What made 
this problem particularly challenging was that it was an intermittent production issue that couldn't be reproduced 
consistently, so it required systematic logging, careful analysis of WebSocket behavior, and understanding of distributed 
state management using Redis.
